## 핵심 피드백 요약

### 🔴 CRITICAL — 연구 유형 분리 필수

> **"레트로스펙티브하고 프로스펙티브하고 이게 굉장히 다르다. 이 두 가지를 나눠서 가는게 맞지 않을까?"**

Yohan이 인터뷰 내내 가장 강조한 포인트. 두 유형은 프로세스, IRB 난이도, 데이터 접근 방식이 완전히 다름.

| 구분 | Retrospective (후향적) | Prospective (전향적) |
|------|----------------------|---------------------|
| **데이터** | 이미 존재하는 병원 내 데이터 사용 | 새로 참여자로부터 수집 |
| **IRB 난이도** | 상대적으로 쉬움 ("어지간해서는 막 건드는 거 같지 않던데") | 매우 어려움 (사람에게 위해 가능성) |
| **참여자 모집** | 불필요 (데이터가 이미 있음) | 필수 (대상자 선정, 동의 등) |
| **현실** | Yohan 연구실은 대부분 이 유형 | 임상의사 참여 필수, 프로세스 복잡 |
| **플랫폼 필요도** | 시각화/분석 도구 위주 | 전체 연구 설계 + IRB + 모집 |

---

### 🟡 IMPORTANT — 연구 프로세스는 선형적이지 않다

> **"이거가 순서가 딱 일직선으로 쭉 가는게 아니라 진짜 오락가락 막 뒤죽박죽으로 섞이고 난리가 난리도 아니다"**

- 현재 6스텝 흐름(AI인터뷰 → 연구설계 → 데이터수집 → 타겟 → IRB → 에스크로)은 **현실과 괴리**
- 진행 중인 연구도 6개월째 대조군/실험군 구성이 계속 바뀌고 있음
- IRB를 **최대한 넓게 잡아서** 내부 변경에도 커버될 수 있게 작성함
- **시사점:** 선형 스텝 대신 유연한 수정/반복 구조 필요

---

### 🟢 POSITIVE — IRB 자동화는 확실한 페인포인트

> **"IRB 쓰는 거 도와주는 거는 진짜 도움이 될 거 같다"**
> **"굉장히 행정적으로 이건 꼭 해야 되긴 하는데 진짜 정말 귀찮은"**

- 1순위까지는 모르겠지만, 연구자들이 진짜로 귀찮아하는 것은 확실
- 헬스케어 연구에서 **규제가 가장 골치 아픈 부분** → 여기서 차별점 가능
- 규제 기반 연구 설계 가이드 제공이 경쟁력이 될 수 있음
- 단, 연구 설계 자체는 젠스파크 등 경쟁자 많음 → **규제/IRB에 특화**해야 차별화

---

### 🟢 POSITIVE — 데이터 시각화는 바로 쓸 수 있는 가치

> **"시각화해서 보여주면 좋죠. 바로바로 시각화해서 볼 수 있으면"**
> **"매번 뭘 볼지 결정하고 파이썬 코드를 짜서… 이 과정 자체가 진짜 굉장히 귀찮은"**

- 현재 연구 데이터 확인: CSV 가져와서 Python 코드로 시각화 → 매우 귀찮음
- **참여자별 상세 패널(72시간 CGM, 스파크라인)** → "괜찮은 아이디어, 좋은 거 같아"
- 먼저 보여줄 것: **시계열 그래프 + 히스토그램(빈도)** → 연구자가 가장 먼저 확인하는 것
- **단, 주의:** 현재 와이어프레임이 **특정 데이터(혈당/위고비)에 너무 커스터마이징** 되어 있음
- 다양한 데이터 유형이 들어와도 동일한 시각화가 가능해야 함

---

### 🔴 CRITICAL — 데이터 마켓플레이스 수요 불확실

> **"병원 안에 소속된 사람들은 병원 데이터를 접근 가능하고 그걸 연구를 하기 때문에 퍼블릭 데이터를 쓰는 거에 약간 굳이…"**

- 병원 소속 연구자 → 원내 데이터 접근 가능 → **외부 데이터 플랫폼 필요 없음**
- Yohan 본인도 EMR 데이터로 연구하므로 외부 데이터 불필요
- **수요가 있을 수 있는 대상:** 병원 밖 헬스케어 스타트업, 건기식 기업 등
- 1차 미팅의 "외부 데이터 필요" 피드백은 → 공학계열 저널 제출 시에만 해당

---

### 🟡 IMPORTANT — 과거 데이터 판매 모델은 가능성 있음

> **"내가 당뇨병 환자야. 내 기록이 있어. 그 부분을 갔다 팔 수 있다. 이렇게 한다고 하면 괜찮아지긴 하겠죠."**

- 사용자가 자신의 **과거 건강 데이터(웨어러블, 건강보험 등)를 판매**하는 모델
- 이미 수집된 데이터 → 사람에게 위해 없음 → IRB 부담 대폭 감소 가능성
- 웨어러블(애플 건강, 삼성헬스) 과거 데이터 가져오기 가능
- **반드시 법적 검토 필요** — 변호사 미팅에서 확인해야 함
- 사용자 익명화(ZKP) → IRB 통과 가능 여부도 확인 필요

---

### 🟡 IMPORTANT — 플랫폼 스코프가 너무 넓다

> **"이 두 가지(연구 설계 + 데이터 시각화)가 각각이 해비한 태스크… 회사 두 개로 쪼개도 될 거 같은데"**

- 연구 설계 도구 + 데이터 시각화/분석 플랫폼 = 사실상 회사 2개 분량
- **하나의 페인포인트를 명확하게 잡아서 깊게 해결하는 방향**이 맞음
- 김정연도 동의: "진짜 너무 뭐가 많다, 명확한 하나에만 집중해서"
- 확장은 가치 입증 후에

## 예상치 못한 인사이트

### 1. Yohan은 곧 병원을 떠남
> **"나는 곧 뜰 거야"** / **"이쪽이 너무 보수적이고 경직돼 있는 느낌"** / **"연구 자체가 재미가 없어"**

- 증강현실/HCI 출신 → 의료 분야 경직성에 불만
- 매주 토요일 다른 랩(3D 매핑) 참여 예정
- 시사점: 추가 인터뷰어 확보 필요 (Yohan 이후 연결 끊길 수 있음)

### 2. 의사 중심 의사결정 구조
> **"MD가 아닌 이상은 좀 인정을 안 해주는 그런 것도 있기도 하고"**

- 비의사 연구자의 혁신 제안은 잘 받아들여지지 않음
- 플랫폼이 의사 PI를 직접 타겟하지 않으면 도입이 어려울 수 있음

### 3. 병원은 돈은 많지만 혁신은 느림
> **"돈은 많고… 절대 안 망하지만 이 분야는"** / **"적당히 돈 받으면서 공무원처럼 살고 싶은 사람한테 최고"**

- 예산 확보는 상대적으로 수월하지만 변화 저항이 큼
- 파일럿 도입 시 "혁신"보다 "효율화/비용절감" 프레이밍이 필요

### 4. OMOP CDM 표준 확인
- 서울대병원은 원내 데이터를 OMOP CDM으로 포팅 중 (5-6년째)
- 다양한 테이블(Person, Concept, Condition, Drug 등) + 필드 구조
- 데이터 시각화 시 이 구조를 참고해야 함
- **MIMIC** 데이터셋도 참고 추천 (상업적 이용은 불가)

### 5. 전남대병원 CD 스캐너 해킹 사례
- CD 스캐너를 통해 바이러스 침투 → 메인넷 감염
- 개인 의료영상(CT/MRI) 클라우드 보관 서비스 → 잠재적 기회
- 하지만 "뚫어야 할 벽이 크다" (Yohan 의견)

## 1차 vs 2차 인사이트 비교

| 주제 | 1차 미팅 (2/10) | 2차 미팅 (2/13) | 변화 |
|------|----------------|----------------|------|
| **IRB** | "원스톱이 킬링 포인트" | "도움은 되지만 1순위까지는 모르겠다" | ⬇️ 기대치 조정 |
| **데이터 반출** | "병원 데이터 반출 불가" | "병원 소속이면 원내 데이터 접근 가능" | ↔️ 맥락 구체화 |
| **B2C 관심** | "데이터 민주화? 관심 없다" | "과거 데이터 판매는 가능할 수도" | ⬆️ 새 가능성 |
| **참여자 모집** | "어렵지 않다" | "레트로는 불필요, 프로는 의사 필수" | ↔️ 유형 분리 |
| **CDM 변환** | "몇십 년째 진행중" | OMOP CDM 구조 직접 공유 | ↔️ 구체화 |
| **플랫폼 범위** | 다방면 탐색 | "회사 두 개 분량, 하나만 해라" | 🔴 범위 축소 권고 |
| **FL/클린룸** | 관심 표명 | "먼저 다른 거부터 해결하시고" | ⬇️ 후순위 |
