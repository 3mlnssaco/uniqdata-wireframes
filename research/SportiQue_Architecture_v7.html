"환자의 몸에서 나온 데이터, 환자의 지갑, 환자의 동의는 Core. 그 외 연구/기관 워크플로우는 UniQLab Backend."

3계층 ID (재식별 불가): Layer 1: Wallet Address → core_auth에만 (uniqdata_db에도 없음) | Layer 2: person_id (UUID) → uniqdata_db.uq_cdm 골든테이블 FK | Layer 3: participant_number → uniqlab_db.research (연구자가 보는 유일한 ID, 프로젝트마다 다름)

🏗️ 서버별 상세 역할
🟢 uniqdata-core
FastAPI (Python) · Port 8000(default) · Core Team 담당
  • Auth (Passkey/PRF)
    WebAuthn Level 3, PRF 키 파생, JWT 발급/갱신, JWKS 엔드포인트 제공
  • Data Input ⭐
    건강 데이터 수신 → 검증 → AES-256-GCM 암호화 → uniqdata_db.uq_cdm (골든테이블) 저장 → XRPL 앵커링
  • Consent ⭐
    동의서 생성/서명(ed25519)/범위 관리/철회 + XRPL CONSENT_GRANT/REVOKE 앵커링
  • XRPL / UQD
    EscrowCreate/Finish/Cancel, UQD 관리, DATA_INPUT/DATA_ACCESS 앵커링
  • Notification
    푸시 알림 통합 관리 (Core + UniQLab Backend 요청 모두)
  • Internal API
    실코드 기준 13개 엔드포인트: /api/v1/internal/matching/*, /tier1/*, /token/*, /escrow/*, /health
📊 uniqlab-backend
Spring Boot 3.2 (Java 17) · Port 8080 · 백엔드팀 담당
  • Research ⭐
    프로젝트 CRUD, 마일스톤 추적, 참여자 매칭, 적격 기준, 데이터 요청
  • IRB ⭐
    제출/심의/문서 관리/워크플로우 히스토리, AI 문서 생성(Claude API)
  • Settlement ⭐
    보상 계산, settlement_batches/items 생성 → Core에 EscrowFinish 요청
  • UniQLab
    기관 등록/관리, 구성원 관리, OWNER/PI/IRB_MEMBER/STAFF 역할
  • Synthetic/FL (Phase 2)
    합성데이터셋 생성, FL 라운드 관리
  • Core Push 수신
    Core → /api/uniqlab/project/{id}/participant, /dataset(라이브), /dataset/batch, participant DELETE, /settlement(목표)

🔒 인증/보안 계층
[인증 흐름]
1. 사용자 → 🟢 Core: Passkey 인증 (WebAuthn Level 3 + PRF Extension)
2. 🟢 Core → 사용자: Access Token (JWT, 15분) + Refresh Token (7일)
3. 사용자 → Gateway: Authorization: Bearer {access_token}
4. Gateway → 🟢 Core 또는 🔵 UniQLab Backend: 요청 전달

[JWT Claim 계약 — v7 확정]
🟢 Core 발급 JWT:
  { "sub": "person_uuid", "sid": "session_uuid",
    "roles": ["user"],         ← v7 추가 (연구자: ["user","researcher"])
    "email": "user@example.com", ← v7 추가
    "exp": 1707436800, "type": "access" }

🔵 UniQLab Backend JwtAuthFilter:
  roles = claims.get("roles", List.class);
  if (roles == null) roles = List.of("user");  ← v7 null 방어
  email = claims.get("email", String.class);    ← nullable 허용
  wallet → 미발급 (지갑주소 Core 밖 노출 금지 절대 규칙)

[JWT 검증]
🟢 Core:     직접 검증 (signing key 보유)
🔵 UniQLab Backend: 공유 시크릿으로 검증 (jwt.secret 환경변수)

[Core Push 수신 인증 — v7 추가]
🟢 Core → 🔵 UniQLab Backend Push:
  헤더: X-Service-Auth (= CORE_PUSH_TOKEN 환경변수)
  SecurityConfig: /api/uniqlab/project/** → permitAll (컨트롤러에서 토큰 검증)
  재시도: enabled=true, delay=300s, batch=20, max=3

❓ FAQ
Q: 분리 기준이 "민감/비민감 데이터"인가?
A: 핵심 기준은 "환자에 대한 정보 vs 활동에 대한 정보".
사람 자체에 대한 것(건강, 신원, 키, 동의, 지갑) → Core
사람이 참여하는 활동에 대한 것(프로젝트, IRB, 정산, 기관) → UniQLab Backend
Q: uniqdata_db에는 총 몇 개 테이블?
A: 전면 개편 (3 Schemas). core_auth(지갑), uq_cdm(OMOP 골든테이블 19개), uq_mapping(매핑테이블 4개)로 구성됩니다.
Q: uniqlab_db에는 총 몇 개 테이블?
A: 연구 프로젝트 데이터 중심 (research). 6 골든테이블을 사용하여 participant_number 기반으로 데이터셋이 구성됩니다.
Q: IRB 관련 테이블은 어디에?
A: 🔵 uniqlab_db에 4개: irb_submissions, irb_documents, irb_review_comments, irb_workflow_history. IRB는 연구 "활동"이므로 UniQLab Backend 담당.
Q: 정산은 어디서?
A: 정산 계산(대상자 산정, 금액 계산, settlement_batches 생성) → 🔵 UniQLab Backend
정산 실행(XRPL EscrowFinish, UQD 크레딧 적립) → 🟢 Core
UniQLab Backend가 POST /api/v1/internal/escrow/finalize로 요청 → Core가 실행 → tx_hash 반환
Q: Internal API와 Core Push 갯수는?
A: UniQLab Backend→Core Internal API: 13개 (REST, X-Service-Auth 인증)
Core→UniQLab Backend Push API: 5개 (현재 2개 라이브, 3개 목표)
