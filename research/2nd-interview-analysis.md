## 1. 블록체인 피봇 배경 및 데이터 소유권

### 피봇 결정 배경

- 국내 블록체인 분야의 현실적 어려움 인식
- **핵심 발견:** 사용자들이 자신의 데이터 소유권이나 활용 방식에 대해 큰 관심이 없음
- 약관 동의를 꼼꼼히 읽지 않고 "다음" 버튼을 누르는 행태가 보편적
- 그러나 데이터 유출 사건 발생 시 "내 데이터 돌려내라"는 모순적 반응

### 데이터 프라이버시 우려

- 보험사가 개인 건강 데이터로 알고리즘을 만들어 보험료를 자동 책정할 가능성
- 데이터 출처를 명확히 하고 필요한 곳에만 쓰이게 하려는 것이 원래 목표

### Web 3.0 개념 연결

- 플랫폼은 Web 2.0(회사가 데이터 보유)과 Web 3.0(개인이 데이터 소유) 경계에 위치
- Web 3.0: 개인이 지갑에 데이터를 보유하고, 로그인 시에만 일시적으로 공유 후 회수
- 플랫폼에 개인정보를 넘기지 않는 구조 지향

## 2. 와이어프레임 시연 — 연구 설계 플로우

### 시연된 3단계 플로우

1. **연구 설계:** 연구자가 자연어로 목적 입력 → AI가 대화형으로 연구 설계 초안 자동 생성
2. **IRB 판정:** 연구가 IRB 필요/면제/불필요 중 어디에 해당하는지 분석
3. **연구 진행:** 데이터 수집, 실험군 배치, 보상 지급까지 일괄 관리

### 주요 기능 요소

- **채팅 UI:** IRB 심사 요건에 필요한 7가지 연구 요소(주제, 대상, 개입 등)를 구조화된 질문으로 파악
- **문헌 검색:** PubMed에서 최신 논문 자동 가져오기 (IRB 심사가 최신 연구 자료를 요구하므로)
- **데이터 수집 계획:** 웨어러블 데이터, 사용자 직접 참여 데이터, 정성 데이터(사진 등) 선택
- **실험군 설정:** 사전 타겟 추천 + 추가 인원 랜덤 배치
- **보상 시스템:** 프로젝트 지갑에 돈 예치 → 사용자 데이터 입력 시마다 자동 보상 지급 (에스크로)

## 3. 연구 프로세스 피드백 — 핵심 지적사항

### ⚠️ 연구 프로세스는 비선형적이다

> "순서가 딱 일직선으로 쭉 가는 게 아니라 진짜 오락가락 뒤죽박죽으로 섞인다"

- 3단계로 한 번에 완료하는 방식은 **현실과 맞지 않음**
- 코호트 연구 사례: 6개월 동안 연구 내용이 계속 변경됨
- 대조군/실험군 설정도 연구 진행 중 변경되는 경우 빈번
- 초기 설정 → 중간 수정 → 다시 초기로 돌아가는 반복이 일상적

### ✅ IRB 행정 자동화는 긍정적

> "IRB 쓰는 거 도와주는 거는 진짜 도움이 될 거 같다"

- IRB 작성 등 행정적 부분은 연구에서 가장 귀찮은 일 중 하나
- 행정 자동화 방향성 자체는 연구자들에게 유용할 것
- 다만 전체 연구 프로세스 지원이 실제로 얼마나 도움이 될지는 불확실

### IRB는 넓게(Broadly) 작성하는 것이 관행

- IRB 작성 시 최대한 넓게 잡아서, 연구 내용이 바뀌더라도 커버되도록 함
- 데이터셋 자체를 바꾸는 것이 아니라, 기존 데이터 안에서 기준을 바꾸고 조정하는 것은 같은 IRB 내에서 커버 가능
- **플랫폼 설계 시사점:** IRB를 좁게 자동 생성하면 오히려 재심사 빈도가 높아질 수 있음

## 4. Retrospective vs. Prospective — 핵심 구분

### ⚠️ 이 구분이 플랫폼 설계의 근본적 분기점

> "이 두 가지가 사실 굉장히 달라요. 연구하는 방향이나 프로세스도 굉장히 다르다"

| 구분 | Retrospective (후향적) | Prospective (전향적) |
|------|----------------------|---------------------|
| **데이터** | 이미 존재하는 데이터 활용 | 사람에게서 새로 수집 |
| **IRB** | 상대적으로 쉬움 | 매우 까다로움 |
| **프로세스** | 단순 (데이터 있음 → 승인 → 분석) | 복잡 (대상자 모집, 동의서, 추적관찰) |
| **난이도** | 낮음 | 높음 |
| **비유** | 과거 기록을 보는 것 | 실험 대상자를 모집하고 추적하는 것 |

### 연구자 조언

- 이 두 가지를 **반드시 나눠서** 고민하고 개발해야 함
- 인터뷰 대상 연구자에게 어떤 유형의 연구를 하는지 먼저 파악한 후 조언을 구해야 함
- 해당 연구실은 Retrospective 연구가 **압도적으로 많음**
- Prospective 연구는 임상 의사 없이는 거의 불가능

## 5. 데이터 시각화 피드백

### ⚠️ 특정 데이터에 종속된 UI

> "현재 UI가 혈당 데이터 보기에 맞춰져 잘 조정되어 있지만, 다른 종류의 데이터가 들어왔을 때도 동일하게 표시할 수 있을지 의문"

- 현재 시각화가 **혈당 중심**으로 커스텀되어 있다는 인상이 지속적으로 언급됨
- 혈액 데이터, 혈압, 영상 데이터 등 **컬럼 속성이 매우 다른 데이터**를 수용할 수 있어야 함
- 데이터 플랫폼이라면 다양한 데이터에 대해 **동일한 가시성**을 보여줄 수 있어야 함
- 수치 외에 정성적 데이터도 존재하므로 수치 중심 테이블에는 한계가 있음

### ✅ 시각화의 긍정적 가치

> "코드를 짜서 데이터를 올리고 봐야 하는 과정을 생략할 수 있으므로 확실히 좋다"

- 연구실에서는 CSV를 파이썬 코드로 직접 분석 → 이 과정이 매우 번거로움
- 데이터를 클릭했을 때 평균이나 추이를 **바로 시각화**해서 볼 수 있는 기능은 매우 유용
- 시계열 데이터는 기본 1D 그래프, 카테고리컬 데이터는 빈도수/히스토그램 등 **데이터 타입별 자동 시각화** 제안

### 참고 데이터 표준 및 데이터셋

- **OHDSI CDM (Common Data Model):** 대학병원들이 EHR 시스템 구축 시 따르는 표준. CSV 형태로 Person, Concept, Condition, Drug 등 다양한 테이블로 구성
- **MIMIC 데이터셋:** AI 분야에서 많이 사용되는 공개 병원 데이터셋. EMR 데이터 시각화 개발 시 참고 권장
- 해당 대학병원은 원내 데이터를 CDM 형식으로 포팅하는 작업을 **5~6년간** 진행 중

## 6. 익명 사용자-연구자 매칭 — IRB 문제

### ⚠️ 익명 매칭 시 IRB 통과 불확실

> "익명의 사용자들하고 연구자하고 연결시켜 버리면 피실험자가 누구인지 모르는 문제가 발생"

- IRB는 **피실험 대상자가 명확해야** 하며, 개인정보 동의서를 받아야 함
- 익명으로 연결하면 이 요건을 충족하지 못할 가능성
- 이 모델은 **임상 연구보다 Amazon Turk** 같은 익명 연구 쪽에 가까움
- 영지식 증명(Zero-knowledge proof)으로 질환 정보는 공개하되 신원은 보호하는 접근은 방향성은 맞으나, **IRB 통과 여부는 법적 확인 필요**
- → 변호사 미팅을 통한 법적 검토 예정

---

## 7. 플랫폼 범위 — "두 가지 Heavy Task" 문제

### ⚠️ 연구 설계 + 데이터 시각화 = 과부하

> "IRB를 만들고 연구를 설계하는 부분과, 데이터를 시각화해서 보여주는 부분이 각각 매우 방대한(heavy) 작업이다"

- 이 두 영역은 각각 **전문 도구**(예: Tableau)가 존재할 정도로 방대함
- 하나의 플랫폼에서 동시에 해결하려는 것은 **과부하**
- **인과관계 맵 구축**도 그 자체가 이미 하나의 연구 주제에 해당하는 방대한 작업
- 단순히 랜덤 포레스트나 상관관계로 중요도를 나열하는 방식으로는 변수의 의미를 파악할 수 없음

## 8. MVP 전략 조언

### 핵심 메시지: "좁고 깊게"

> "타겟을 작게 잡아서 명확한 고충점을 해결하여 가치를 인정받고 확장하는 방식이 일반적"

### 추천 방향: 규제/행정 해결에 집중

> "헬스케어 분야의 핵심인 '규제' 문제를 해결하여 경쟁력 확보"

- 연구 설계 지원만으로는 **이미 경쟁자가 많음**
- 연구자들이 잘 모르는 영역: 어떤 데이터를 수집할지, 규제에 걸릴지, IRB를 어떻게 작성해야 하는지
- **한국의 업데이트된 법규에 따른 가이드**를 제시하는 행정 지원이 차별화 포인트
- IRB 행정 업무는 연구 아이디어 창출에 직접적 도움은 안 되지만, **반드시 해야 하는 귀찮은 업무**로 고충이 큼

## 9. 헬스케어 시장 특성

### 시장 환경에 대한 현장 인사이트

- **극도로 보수적이고 경직된** 시장
- **의사(MD) 중심** 구조 — MD가 아니면 연구에서 인정받기 어려움
- 자본력은 풍부하여 연구 지원은 확실하지만, 혁신을 이루려는 사람에게는 부적합한 환경
- "만년 공무원 같은 신세" — 비MD 연구자의 체감
- 분야 자체는 **절대 망하지 않을 것**이라는 확실성은 있음

## 10. 병원 연구자의 외부 데이터 플랫폼 수요

### ⚠️ 병원 연구자는 외부 데이터가 필요 없을 수 있음

> "대부분의 연구자들은 이미 병원 소속으로 원내 데이터에 접근 가능하기 때문에 외부 데이터 플랫폼에서 추가적인 데이터를 가져올 필요가 없을 수도 있다"

- 해당 연구자 본인도 EMR 데이터, 병원 내 환자 데이터를 사용 → 외부 플랫폼 불필요
- 외부 검증 데이터에 대한 수요는 **논문 게재 저널에 따라 다름**: 임상지는 거의 없고, 공학계열 저널/컨퍼런스에서 태클이 많음

### 제안된 대안 타겟

- **건강기능식품 회사** — 자체 데이터 인프라 부재
- **헬스케어 스타트업** — 데이터 접근이 어려운 조직
- **병원 외부 연구자** — 원내 데이터 접근 불가
- → 이들을 대상으로 **데이터 수집 및 접근성이 메리트가 되는지** 추가 인터뷰 필요

## 11. 데이터 풀 vs. 데이터 구독 — 분리 설계

### 두 가지 연구 페이지로 분리

| 구분 | 데이터 풀 (Retrospective) | 데이터 구독 (Prospective) |
|------|--------------------------|--------------------------|
| **성격** | 과거 데이터 불러와 분석 | 매일 데이터 입력받아 추이 관찰 |
| **IRB** | 불필요 또는 간소 | 심의 필요 |
| **UI** | 데이터 시각화 + AI 분석 환경 | 연구 개요 테이블 + 시계열 추적 |
| **데이터 출처** | 축적된 웨어러블/건강 데이터 | 실시간 사용자 입력 |

## 12. 추가 논의 사항

### 영상 데이터 관리

- CT, MRI 등 영상 데이터는 개인이 CD/USB로 소지 → "데이터가 떠다니는" 문제
- 클라우드 기반 영상 데이터 보관 서비스 구상
- 보안 사례: 병원 CD 스캐너 해킹으로 전산망 감염 사례 존재

### 클린룸(Clean Room) 환경

- A 병원과 B 병원이 데이터를 올려 공동 연구할 수 있는 클라우드 환경
- 아이디어 자체는 좋으나, **현재 단계에서는 급선무가 아님** — 다른 것부터 해결 후 진행 권장

### 연구 계획서 기반 칼럼 재배치

- 연구 계획서 작성 후, 데이터 칼럼을 연구 연관도에 맞게 자동 재배치하는 아이디어 → 긍정적 반응

### 법적 검토 필요 사항

- 익명 사용자-연구자 매칭 시 IRB 통과 가능 여부
- 사용자가 자발적으로 데이터를 판매할 때의 법적 문제
- 웨어러블 데이터(Apple Health, Samsung Health 등) 판매 시 법적 쟁점
- → **변호사 미팅 예정** (기술 문서, 샌드박스 관련 전문)

## 핵심 Action Items

1. **Retrospective / Prospective 분리 설계** — 플랫폼 구조를 이 구분에 맞춰 근본적으로 재설계
2. **데이터 시각화 범용성 확보** — 혈당 중심 UI를 다양한 데이터 타입을 수용하는 구조로 전환 (OHDSI CDM, MIMIC 참고)
3. **MVP 범위 축소** — 연구 설계 + 데이터 시각화를 동시에 하지 말고, **IRB 행정 자동화** 하나에 깊게 집중
4. **타겟 재검토** — 병원 연구자 외에 건강기능식품 회사, 헬스케어 스타트업 등 데이터 접근이 어려운 대상 인터뷰
5. **법적 검토** — 익명 매칭 IRB 통과 가능성, 데이터 판매 합법성 확인 (변호사 미팅)
